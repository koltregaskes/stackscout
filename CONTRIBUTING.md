# Contributing

Thanks for helping with `stackscout`.

## Principles

- keep the public catalog data accurate and well-sourced
- keep the build static-first and dependency-light
- prefer clear editorial quality over noisy automation
- do not commit secrets, machine-local paths, or private workspace notes

## Workflow

1. update or add the relevant catalog / tool / collection data
2. run the site locally and verify the affected pages render correctly
3. verify the public build contains only public-safe data
4. commit only the intended scope

## Pull requests

- describe what catalog area changed (e.g. `tools`, `categories`, `collections`, `radar`)
- call out any data-source or licensing changes explicitly
- include the exact verification commands you ran
